### 영양리포트>일간채움
'영양 리포트(Sub 2)' 화면의 기존 UI(제공된 스크린샷)는 B2B 의료 데이터 베이스처럼 건조하며, 부모가 느끼는 '죄책감(0%로 표시된 영양소)'과 '실행 지침(텍스트의 나열)'이 완전히 분리되어 있다는 치명적인 결함이 있습니다.

이를 해결하기 위해 '식이 브릿지(Dietary Bridge)' 모델을 도입하여 코드를 전면 재설정했습니다. 우측 Canvas에서 동작하는 코드를 먼저 확인하십시오.

1. [Assumptions] (작업 가설)
[ASSUMPTION] 사용자는 아이가 먹은 모든 음식(이유식, 반찬)을 앱에 매일 기록하지 않습니다. 이를 요구하는 순간 앱의 리텐션은 붕괴합니다.

[ASSUMPTION] 영양제만으로 권장량(RI)을 100% 채우는 것은 불가능하며 바람직하지도 않습니다. 현재 스크린샷의 UI는 영양제만으로 0%인 상태를 '실패'처럼 보이게 만들어 부모에게 불필요한 불안을 조장합니다.

[ASSUMPTION] 과거 데이터 트래킹은 복잡한 꺾은선 그래프보다, 직관적인 '주간 달력(Weekly Tracker)' 형태가 모바일 UX에 적합합니다.

2. [Core Reasoning] (재구성 논리)
기존 화면의 벽돌 같은 텍스트 가이드(image_dbc11d.png)와 차트(image_dbc145.png)를 단일 시각화 컴포넌트로 병합했습니다.

목표(RI)와 상한(UL)의 시각적 분리: 차트의 100% 기준을 상한선(UL)이 아닌 **권장량(RI)**으로 맞췄습니다.

식이 브릿지 시각화: 건기식으로 채운 양은 '실선(Solid)'으로, 부족한 양은 '점선(Dashed)'으로 표기합니다. 그리고 그 점선 빈칸 안에 정확히 '무엇을 먹여야 채워지는지(예: 🥛우유 1컵)'를 시각적 블록으로 직접 삽입했습니다.

이를 통해 유저는 "우리 아이 칼슘이 0%네, 어떡하지?"가 아니라, **"건기식으론 안 먹였지만, 오늘 치즈 2장 먹였으니 완벽하게 채웠네"**라는 심리적 안도감을 즉각적으로 얻게 됩니다.

3. [Risks & Blind Spots] (리스크 및 맹점)
식품 데이터의 파편화 (Data Fragmentation): "우유 1컵"이라는 기준이 모호할 수 있습니다(100ml vs 200ml). 식품 DB를 연동할 때 1회 제공량(Serving Size)에 대한 명확한 내부 기준과 UI 툴팁이 필요합니다.

알레르기 변수 (Allergen Conflict): 부족한 영양소를 채우기 위해 제안한 식품(예: 우유, 계란, 견과류)이 해당 아이의 알레르기 유발 물질일 경우 심각한 신뢰도 하락이 발생합니다.

4. [Strategic Implications] (전략적 함의)
이 UI는 VitaKids를 단순한 '영양제 성분 분석기'에서 **'일간 육아 식이 코파일럿'**으로 격상시킵니다.

영양제로 부족한 부분을 일반 식품으로 치환해 보여주는 기능은 경쟁 앱들이 가지지 못한 강력한 **해석적 가치(Interpretive Value)**를 제공합니다. 이는 유저가 매일 저녁 아이를 재우고 "오늘 식단이 어땠나"를 복기하기 위해 앱을 열게 만드는 트리거가 됩니다.

5. [Immediate Actions] (즉각적 실행 과제)
우측 캔버스의 상단 주간 캘린더 요일을 클릭하여, 과거의 기록과 오늘의 상태가 어떻게 전환되는지 테스트하십시오. (목, 금, 토요일에 각각 다른 더미 데이터가 바인딩되어 있습니다.)

'칼슘' 항목의 점선 영역에 제안된 [음식 블록] UI가 기존 텍스트 나열 방식보다 얼마나 인지 부하를 낮추는지 확인하십시오.

알레르기 필터링 로직 도입을 위한 User Profile DB 설계를 즉각 착수하십시오.


### 영양리포트>성장추이
1. [Assumptions] (작업 가설)
[ASSUMPTION] '미래의 섭취량'은 예측 불가능합니다. 따라서 미래 차트는 "현재 먹이고 있는 조합을 그대로 유지할 경우, 아이의 성장에 따라 변경되는 권장량(RI) 대비 얼마나 부족해지는가(Gap)"를 보여주는 **'결핍 예측 시뮬레이션'**으로 작동해야 합니다.

[ASSUMPTION] 일간 데이터(미시적 채움)와 월별 시계열 데이터(거시적 추세)를 한 화면에 위아래로 나열하면 인지 과부하가 옵니다. 상단에 Segmented Control(탭)을 두어 **[일간 리포트]**와 **[성장 추이]**로 화면을 명확히 분리했습니다.

[ASSUMPTION] 무거운 외부 차트 라이브러리(Chart.js 등) 사용은 모바일 웹 환경에서 렌더링 지연을 유발합니다. 브라우저 네이티브인 SVG를 활용해 가볍고 직관적인 커스텀 스파크라인(Sparkline) 차트를 직접 구현했습니다.

2. [Core Reasoning] (재구성 논리)
시계열(Macro)과 당일(Micro)의 분리: 사용자의 방문 목적은 "오늘 뭐 먹이지?"(일간)와 "지금까지 잘 먹였나? 앞으로 뭐 사야 하지?"(추이) 두 가지로 나뉩니다. 이를 상단 탭으로 분리하여 의도에 맞는 화면만 제공합니다.

미래 결핍의 시각화 (The Future Gap): * 과거~현재: 기록된 섭취량(실선)과 과거의 권장량(점선)을 교차하여 보여줍니다.

미래: 아이가 성장함에 따라 권장량(RI) 선이 계단식으로 상승합니다. 반면 섭취량 선은 현재 상태로 평행하게 이어집니다. 이 둘 사이에 발생하는 시각적 '빈 공간(Gap)'이 부모에게 "다음 단계 영양제를 준비해야겠다"는 강력한 행동 촉구(Call to Action)로 작용합니다.

기준선(RI/UL)의 직관적 인지: 차트의 배경을 영역별로 분리했습니다. 상한선(UL) 부근은 붉은색 경고 영역으로, 권장량(RI) 부근은 안정적인 녹색 영역으로 칠해 선의 위치만으로도 상태를 직관적으로 파악하게 했습니다.

3. [Risks & Blind Spots] (리스크 및 맹점)
Y축 스케일 왜곡 (Scale Distortion): 칼슘(2500mg)과 비타민D(30μg)는 단위와 절대값이 다릅니다. 동일한 높이의 SVG 차트에 그릴 때 상한선(UL)을 Y축의 Max로 잡으면, 권장량(RI)이 너무 낮아 섭취량 변화가 바닥에 붙어 보일 수 있습니다.

Mitigation: 차트 구현 시 UL을 무조건 천장으로 잡지 않고, 섭취량과 권장량을 중심으로 동적 스케일링을 적용하되 UL 초과 시에만 천장을 확장하는 로직이 향후 필요합니다. (현재는 시각적 예시를 위해 비율을 조정해 두었습니다.)

빈 데이터 처리 (Missing Data): 과거 기록이 듬성듬성할 경우 꺾은선이 끊어지거나 왜곡됩니다. 기록이 없는 달은 '결측치 처리(보간법 적용 또는 점선 처리)'에 대한 정책이 필요합니다.

4. [Strategic Implications] (전략적 함의)
교차 판매의 당위성 확보 (Cross-sell Justification): "3개월 뒤면 비타민D 권장량이 늘어나 현재 제품으론 부족합니다"라는 메시지를 차트로 증명함으로써, 앱 푸시나 상품 추천이 '광고'가 아닌 '맞춤형 컨설팅'으로 격상됩니다.

LTV(고객 생애 가치) 연장: 1년 이상 장기 데이터를 시각화해 줌으로써, 유저가 앱을 삭제하지 못하게 만드는 강력한 Lock-in 효과(Sunk Cost Fallacy)를 창출합니다.

5. [Immediate Actions] (즉각적 실행 과제)
우측 Canvas 상단의 [일간 리포트] / [성장 추이] 탭을 클릭하여 뷰 전환을 테스트하십시오.

[성장 추이] 탭에서 미래 시점(26개월 -> 36개월)으로 갈수록 권장량(목표 점선)이 올라가면서 발생하는 '결핍 예상 구간'의 시각적 효과를 점검하십시오.

실제 유저 데이터가 비어 있을 때(No Data State) 차트를 어떻게 보여줄 것인지(예: 흐린 회색 선과 "기록을 채워주세요" 툴팁) 기획을 보강하십시오.
