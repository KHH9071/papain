(참고: 아래에서 말하는 "CC"는 Claude code가 제시한 plan을 말함.)

이 분석/계획안은 방향은 맞고, 범위 통제도 괜찮다. 다만 그대로 실행하면 “UX 정리”는 되지만 파파인의 핵심 포지션까지는 못 올린다. 이유는 CC가 문제를 잘 짚었지만, 아직 제품 차별점과 리스크 경계를 충분히 계획에 반영하지 못했기 때문이다. 

[Assumptions]

[ASSUMPTION] 네 목표는 “예쁜 개편”이 아니라, 현재 코드 구조를 크게 깨지 않으면서 제품 의도가 더 선명하게 드러나는 개편이다.

[ASSUMPTION] 이번 턴에서는 전면 구현 지시가 아니라, CC 계획안을 채택/수정/보류로 나눠서 다음 단계 프롬프트를 정교화하는 게 맞다.

[ASSUMPTION] 파파인의 핵심 가치는 세 가지다.
    1) 시기별 전체 관점, 2) 유명제품 ↔ 대안제품 탐색, 3) 의료/처방처럼 보이지 않는 안전한 정보 구조.

[Core Reasoning]
1. 이 계획안의 좋은 점: 문제 진단이 현실적이다

CC는 현재 구조를 HomeClient / SearchClient / record/page 중심으로 파악했고, 도메인 로직은 kdri_data / routine_foods / growth_data / store에 있다고 봤다. 또 DashboardClient.tsx를 완전 미사용 dead code로 판단했고, 큰 방향을 “도메인 로직 보존 + UX 레이어 개선”으로 잡았다. 이건 맞다. 특히 지금 코드베이스에서 대규모 재작성이 아니라 기존 자산을 드러내는 방향으로 가야 한다는 인식은 정확하다. 


2. 채택해야 할 부분: Phase 구조는 합리적이다

Phase 1 = Home 선제형, Phase 2 = Search 맥락 탐색, Phase 3 = 탭 간 연결 강화 순서는 좋다. 현재 가장 큰 문제를 “홈이 선택 전에는 의미가 약하고”, “탐색이 카탈로그처럼 느껴지며”, “탭 간 순환 동선이 없다”로 본 것도 타당하다. 이건 지금 보이는 UX의 핵심 병목과 거의 일치한다. 

3. 하지만 핵심 누락 1: “시기별 전체 관점”이 너무 약하게 해석됐다

CC는 Home 개편을 주로 나이 기반 권장 영양소 요약 카드와 충족/부족/주의 숫자 요약으로 풀었다. 이건 필요하지만 충분하지 않다. 네 초기 기획 의도의 핵심은 단순 RI 카드가 아니라 “지금/다음 시기에 무엇을 챙겨야 하지?”를 부모가 한눈에 훑는 브리핑이다.
즉 Home은 숫자 요약판이 아니라 시기 브리핑 화면이어야 한다.
CC 안은 “선제형”으로 가려는 방향은 맞지만, 아직 브리핑 레이어가 아니라 요약 카드 레이어에 머물러 있다. 이 차이가 크다. 


4. 핵심 누락 2: 파파인의 차별점인 “유명제품 ↔ 대안제품” 축이 Search 핵심으로 승격되지 않았다

CC는 Search를 “부족한 영양소 기반 맥락 탐색”으로 강화하고, 유사 제품 추천을 카드 밖으로 끌어내자고 했다. 방향은 좋다. 하지만 이건 아직 보조 개선이다.
너의 기획 의도상, “유명 제품과 비슷한 성분/기능의 다른 제품 소개”는 부가 기능이 아니라 핵심 가치 제안이다. 그런데 계획안에서는 여전히 유사 제품 기능이 getSimilarProducts()의 노출 위치 문제처럼 다뤄진다. 이건 축소 해석이다. 


즉 Search는 두 개의 주 진입이 있어야 한다.
하나는 부족한 영양소 채우기, 다른 하나는 기존에 아는 제품의 대안 찾기다.
현재 CC 계획은 전자만 강하고, 후자는 약하다.

5. 핵심 누락 3: 법/산업 리스크 경계가 계획에 거의 없다

CC는 Record의 Dietary Bridge를 “의료처럼 보이지 않으면서” 강화해야 한다고 썼다. 하지만 그 이상은 없다. 즉 리스크 인식은 문장 수준에 머물고, UI 설계 원칙으로는 아직 내려오지 않았다. 


이건 위험하다.
왜냐하면 Search를 더 강하게 만들수록, 특히 “부족한 영양소 기반 추천”을 전면에 세울수록 사용자는 쉽게 처방처럼 읽는다.
따라서 구현 전에 최소한 아래는 명시돼야 한다.

일반의약품 / 건기식 / 기타를 같은 추천 문법으로 보여줄지, 분리할지

“추천” 대신 “탐색/비교/유사 제품 보기” 중심 카피를 어디까지 지킬지

리콜/알러지/주의 정보의 우선순위를 어디에 둘지

지금 계획안에는 이 경계가 없다.

6. 핵심 누락 4: fake door 처리 방침이 없다

현재 구조를 3개 기능 탭 + 2개 fake door로 인식한 건 맞다. 그런데 계획은 라운지/마이를 사실상 방치한다. 

이건 사소해 보이지만 제품 인상에 치명적일 수 있다.
왜냐하면 핵심 탭 UX를 다듬어도, 사용자가 fake door를 누르는 순간 **“아직 덜 만들어진 앱”**으로 인식하기 때문이다.

지금 단계에서 라운지/마이를 완성할 필요는 없다.
하지만 적어도 아래 둘 중 하나는 결정해야 한다.

숨긴다

명시적 준비중 상태로 설계한다

“그냥 남겨둔다”는 최악이다.

7. 핵심 누락 5: store를 “건드리지 않는 것”이 오히려 병목일 수 있다

CC는 store.ts를 보존 대상으로 두고, 수정이 없거나 최소일 것으로 봤다. 이건 지나치게 보수적일 수 있다. 

왜냐하면 다음 기능들은 대개 상태 계약이 필요하기 때문이다.

Home에서 부족 영양소 CTA → Search 진입 시 해당 필터 자동 적용

Record에서 부족 영양소 보충 CTA → Search 딥링크

Search에서 “대안 보기” 진입 시 기준 제품 상태 보존

즉 “전역 상태 수정 없음”은 목표가 아니라 결과여야 한다.
필요하면 아주 작게라도 상태 계약을 추가하도록 열어줘야 한다.

[Risks & Blind Spots]
1. Phase 1이 너무 얕으면, Home이 여전히 예쁜 집계판에 머문다

지금 안대로면 Home은 빈 상태를 메우고 숫자 요약을 붙이는 수준까지는 좋아진다. 하지만 이 정도로는 사용자가 “왜 파파인을 열어야 하는지”를 바로 이해하지 못할 수 있다. Home은 “현재 섭취 집계”가 아니라 시기 브리핑 + 현재 상태 + 다음 행동이 같이 있어야 한다.

2. Search 강화가 곧바로 리스크 강화로 이어질 수 있다

부족한 영양소 기반 추천 배너는 전환에는 유리하지만, 잘못하면 “철분 부족 → 이 제품 추천”처럼 너무 직접적으로 읽힌다. 이건 특히 일반의약품이 섞일 경우 위험하다.
따라서 Search는 추천 강화를 하기 전에 정보 분류와 카피 톤이 먼저 정리돼야 한다.

3. 유사 제품 기능이 약하면 앱의 경제적 가치가 안 보인다

부족 영양소 보충만 강조하면 결국 일반 영양앱이 된다.
파파인의 시장성은 “부모가 아는 유명제품을 기준으로 더 넓은 시장을 구조화해준다”는 데 있다. 여기서 우위를 못 만들면 이후 데이터 파이프라인의 가치도 반감된다.

[Strategic Implications]
1. 이 계획안은 “UX 정리안”으로는 합격, “제품 포지셔닝 강화안”으로는 미완성이다

정리하면, CC 안은 실행 가능한 프론트 개편 계획이다. 그래서 버릴 필요는 없다.
하지만 그대로 실행하면 결과물은 아마 “조금 더 똑똑해진 영양 관리 앱”이 될 가능성이 크다.
네가 원하는 건 그보다 더 날카로운 시기 브리핑 + 비교 탐색 엔진이다.

2. 다음 단계는 전면 구현이 아니라, Phase 1 범위를 더 똑똑하게 재정의하는 것이다

지금 바로 Phase 1 구현에 들어가도 된다. 다만 그 Phase 1 정의를 수정한 뒤 들어가야 한다.

내 권장 재정의는 이렇다.

기존 Phase 1: Home 선제형 레이어

수정 Phase 1: Home을 ‘집계판’에서 ‘시기 브리핑 + 현재 상태 요약’으로 전환 + dead code 정리

기존 Phase 2: Search 맥락 탐색

수정 Phase 2: Search를 ‘부족분 탐색’과 ‘대안 탐색’의 2개 진입 구조로 재편

기존 Phase 3: CTA 연결

수정 Phase 3: 탭 연결 + 상태 계약 + Dietary Bridge 시각 강화

즉 순서는 유지하되, 각 Phase의 제품 의미를 강화해야 한다.
